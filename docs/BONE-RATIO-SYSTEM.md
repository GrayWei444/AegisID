# AegisID 骨骼比率系統

> **版本**: 3.0.0
> **日期**: 2026-05-04
> **狀態**: v17 SDK 落地（25 個 2D + 11 個 3D 混合，多基準 + floor-biased 量化）
> **測試頁面**: https://aegisrd.com/face-id-test/

---

## 1. 設計理念

### 1.1 為什麼用骨骼比率

MediaPipe FaceLandmarker 提供 468 個臉部地標點。不同人的骨骼結構不同，表現為地標間距離的**比率**差異。

關鍵設計決策：
- **只用比率，不用絕對距離** — 距離受臉到鏡頭的距離影響，比率不受
- **用 ArcFace 對齊後的座標** — 消除旋轉和平移的影響
- **大量擴充後篩選** — 先系統性覆蓋所有可能的骨骼比率，再通過嚴格測試淘汰不穩定的

### 1.2 唯一 ID 的核心價值

這不是「辨識通過就好」的系統。目標是：

```
同一個人 → 不管什麼時間、光線、裝置、角度 → 永遠產出相同的 ID
不同人 → 永遠產出不同的 ID
```

骨骼比率是天生的物理常數，不會隨環境改變。如果能找到足夠多的穩定比率，
量化後串接成 hash，就能作為唯一身份標識的一部分。

## 2. 比率系統架構

### 2.1 基本流程（v17 SDK）

```
註冊：轉頭 3D 掃描 → MediaPipe 468 landmarks × 多角度
    ↓
真 3D 三角測量（multi-ray least-squares） → 3D landmarks
    ↓
選取 landmark 對 → 計算 2D 與 3D 距離
    ↓
多基準正規化：
  - 垂直距離 → ÷ fh (face height)
  - 水平距離 → ÷ (IPD + browW) / 2
  - 自比率 → 同類距離相除，不需基準
    ↓
floor-biased 量化（BIN_WIDTH=0.25, frac≥0.80 才 ceil）
    ↓
25 個 2D bins → hash2D = SHA-256(bins)
11 個 3D bins → hash3D = SHA-256(bins)
hashCombined = SHA-256(hash2D + hash3D) → 唯一 Face ID

登入：平面刷臉 → 25 個 2D 比率 → bins → 比對本機正面基準 → match rate ≥ 80%
```

### 2.2 Bin 量化（v17 確定版）

```
BIN_WIDTH = 0.25（DEFAULT_BIN_WIDTH）
量化方式 = floor-biased
bin_index = (frac >= 0.80) ? Math.ceil(raw) : Math.floor(raw)
其中 raw = value / BIN_WIDTH, frac = raw - Math.floor(raw)

floor-biased 的原因：
  v14 用 round() 時，bin 邊界在 0.125/0.375/0.625/0.875，
  邊界附近比率（frac 約 0.5）容易跨 bin 跳動。
  v17 改用 frac≥0.80 才進位 — 邊界向後推到 0.80×BIN_WIDTH = 0.20，
  大多數比率自然落在中段，極少跨邊界。

範例：
  ratio = 0.610 → raw = 2.44 → frac = 0.44 < 0.80 → bin = 2
  ratio = 0.595 → raw = 2.38 → frac = 0.38 < 0.80 → bin = 2 ← 同 bin ✅
  ratio = 0.700 → raw = 2.80 → frac = 0.80 ≥ 0.80 → bin = 3 (ceil)

歷史對照：
  v14 BIN_WIDTH=0.25 + round(): 24/24 穩定（單純 3D 比率）
  v17 BIN_WIDTH=0.25 + floor-biased: 3/3 穩定（2D + 3D 混合，多基準）
```

**Bin 邊界問題**：如果一個比率的真實值恰好在 bin 邊界附近（如 0.350），
微小的 landmark 漂移就會導致 bin 跳動。這是篩選不穩定比率的主要原因。

### 2.3 區分力估算

```
假設篩選後保留 N 個穩定比率，每個比率有 ~10 個可能的 bin（0.05 bin width, 比率 0~1）：

N = 20 → 10^20 = 10^20 組合 ✅ 遠超全球人口
N = 30 → 10^30 組合 ✅ 極高區分力
N = 10 → 10^10 ≈ 100 億 ✅ 仍然足夠

但注意：臉部比率之間高度相關，有效獨立維度可能只有 50-60%。
N = 20 的有效空間約 10^12，仍然足夠。
```

## 4. 穩定性測試方法論

### 4.1 測試層級

| 層級 | 條件 | 目標穩定性 |
|------|------|-----------|
| L1 | 頭不動，連拍 5 張 | >95% bin match |
| L2 | 微幅轉頭（±10°）| >90% bin match |
| L3 | 不同拍照距離 | >85% bin match |
| L4 | 不同光線環境 | >80% bin match |
| L5 | 不同裝置/鏡頭 | >75% bin match |

### 4.2 篩選策略

```
Phase 1: 大量擴充（67+ 比率）
    ↓
Phase 2: L1 測試 → 淘汰 <95% 穩定的比率
    ↓
Phase 3: L2-L3 測試 → 再淘汰不穩定的
    ↓
Phase 4: L4-L5 測試 → 最終篩選
    ↓
Phase 5: 保留 20-30 個在所有條件下穩定的比率
    ↓
調整 bin_width → 最大化區分力同時維持穩定性
```

### 4.3 測試結果歷程

**v14（2026-03-20）**：BIN_WIDTH=0.25 + round() → 24/24 純 3D 比率穩定

**v17（2026-03-22）SDK 落地**：多基準 + floor-biased 量化
- 25 個 2D 穩定比率（`STABLE_RATIO_WHITELIST`）
- 11 個 3D 穩定特徵（`STABLE_3D_FEATURES`）
- 同人 3 次重複掃描 hashCombined 一致 ✅
- 已實作於 [sdk/src/face/structuralId.ts](../sdk/src/face/structuralId.ts)

**v15 並行研究（2026-04-16）**：[RESEARCH-REPORT-UNIQUE-FACE-HASH.md](RESEARCH-REPORT-UNIQUE-FACE-HASH.md)
- 5 sessions × 41 captures（基線、多表情、帽子+翹嘴+眼鏡+距離、極端表情、鼓臉頰）
- 26 個跨 sessions 100% bin 一致特徵（offset=0 嚴格篩選 + 黑名單法）
- 41/41 hash 完全一致 ✅
- **跨人區分性尚未驗證**（M1 待辦）
- **未整合進 SDK**（M3 目標 2026-05-15）

## 5. 已知問題與方向

### 5.1 Bin 邊界問題

最大挑戰：比率值恰好落在 bin 邊界時不穩定。

可能解法：
- **加大 bin width** — 犧牲區分力換穩定性
- **使用 bin 中心距離** — 比率離 bin 中心太近的視為不穩定
- **雙 bin 策略** — 同時計算偏移半個 bin 的量化，取共識
- **直接淘汰** — 如果某個比率在測試人群中經常落在 bin 邊界，直接不用

### 5.2 深度路線已判定不可行

**測試數據**: 穩定性 12.3/48 (26%)，等於隨機水平。

**原因**:
1. Depth Anything V2 單次推論不確定性太大
2. Laplacian 曲率場是二階微分，把推論誤差放大
3. 手機端 WASM 推論 ~12.6 秒，記憶體壓力大

**決定**: 放棄深度路線，專注骨骼比率 + pHash 組合。

### 5.3 pHash 作為輔助

- pHash 4×4 (15 bit): 100% exact match，但區分力不足（32768 組合）
- pHash 8×8 (63 bit): 區分力足夠但不穩定（2-4 bit 漂移）
- **策略**: pHash 4×4 作為快速預篩（同一 pHash → 可能同人 → 進一步比對骨骼）
