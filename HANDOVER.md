記錄所有架構決策、設計討論、技術選型理由。

---

## 1. 專案定位

### 1.1 核心理念

**知道是同一人，但不知道是誰。**

AegisID 從 AegisTalk 的 Phase 10（PIN 行為指紋）、Phase 11（設備指紋）、Phase 12（信任分數）、Phase 26（CNN 臉部辨識）中抽離獨立，成為專責身份認證的獨立專案。

### 1.2 為什麼獨立

AegisTalk 是通訊 + AI 防詐。身份認證是跨產品的基礎設施，未來 AegisPay、AegisTrack 都需要身份層。獨立專案可以：

- 獨立版本管理和發布
- 作為 SDK 供多個產品使用
- 隱私設計有明確邊界

## 2. 架構決策記錄

### ADR-001: 身份由 FaceID + PIN 行為聯合構成

**日期**: 2026-03-14

**決策**: 使用 CNN FaceID (512維) 和 PIN 行為指紋 (18維) 共同代表一個人的身份，取聯合信心分數。

**原因**:
- 單獨 FaceID 無法防禦雙胞胎和深偽
- 單獨 PIN 行為無法跨裝置穩定辨識
- 兩者聯合 → 深偽通過 face 但沒有真人 PIN 行為 → 抓到
- 兩者聯合 → 雙胞胎通過 face 但 PIN 行為不同 → 抓到

**取捨**: 需要用戶同時刷臉和輸 PIN，增加一步操作。但這是安全和便利的合理取捨。

### ADR-002: 三表分離 + HMAC 隔離

**日期**: 2026-03-14

**決策**: VPS 使用三張獨立表，不同表使用不同 HMAC secret。

**考慮過的方案**:

| 方案 | 優點 | 缺點 |
|------|------|------|
| 單表全欄位 | 查詢簡單 | 交叉關聯 → 去匿名化風險 |
| 三表分離 + HMAC | 不可跨表關聯 | 查詢需多步 |
| 不存 VPS | 零風險 | 失去防濫用和跨裝置能力 |

**選擇三表分離的原因**:
- face 和 behavior hash 可關聯（都是抽象生物特徵），但不可關聯到 IP/裝置/pubkey
- rate_limits 48hr 自動清除，最壞情況只洩漏 2 天資料
- credit_scores 使用不同 HMAC secret，無法跟 identity_anchors JOIN

### ADR-003: PIN 行為指紋 18 維全比率設計

**日期**: 2026-03-14

**決策**: 移除所有裝置相關絕對值特徵，改為 18 維全比率/模式特徵。

**原因**: 分析現有 20 維特徵後發現，約 10+ 維是裝置指紋（觸控面積、力度、加速度計等），跨裝置完全不同。同一人換手機後 LSH hash 會變 → 無法跨裝置辨識。

**修正**: 只保留人的肌肉記憶特徵（節奏比率、空間偏移模式、習慣），配合固定尺寸鍵盤讓空間特徵也跨裝置穩定。

### ADR-004: 固定 PIN 鍵盤尺寸 280x360px

**日期**: 2026-03-14

**決策**: PIN 鍵盤固定為 280x360px，不隨螢幕縮放。

**原因**: 讓觸控位置的「鍵內偏移」成為人的特徵而非裝置的特徵。同一人在不同手機上按同一個鍵的相對偏移模式相同（肌肉記憶）。

### ADR-005: face_lsh_hash 和 behavior_lsh_hash 可關聯

**日期**: 2026-03-14

**決策**: identity_anchors 表同時存 face 和 behavior hash。

**隱私分析**:
- 可以關聯：face ↔ behavior（都是抽象生物特徵 hash）
- 不可關聯：face/behavior ↔ 真實身份（無 IP、無裝置、無 pubkey）
- 即使攻擊者知道某 face hash 和 behavior hash 屬於同一人，也無法知道這個人是誰

### ADR-006: IP 用完即丟（48hr TTL）

**日期**: 2026-03-14

**決策**: IP hash 只存在 rate_limits 表，48 小時自動清除。

**原因**: IP 對防批量建號有價值（同 IP 24hr 建 20 帳號 → 可疑）。但永久存儲 IP hash 有去匿名化風險（結合 ISP 記錄）。改為短期存儲，用完即丟。

**取捨**: 無法做 3 個月後的歷史追蹤。但詐騙集團批量建號是即時行為，48hr 窗口足夠。

### ADR-007: 只保留 CNN FaceID，廢棄 Landmark 版

**日期**: 2026-03-14

**決策**: 只使用 MobileFaceNet 512 維 CNN embedding，Landmark 128 維降級為 fallback。

**原因**:

| | Landmark 128維 | CNN 512維 |
|---|---|---|
| 跨鏡頭穩定性 | 中 | 高 |
| 抗光線/角度變化 | 低 | 高 |
| 額外模型 | 不需要 | 13MB |
| 計算速度 | <5ms | <50ms |

CNN 版本在所有維度都更強。Landmark 版本保留為 CNN 模型載入失敗時的 fallback。

### ADR-008: VPS 存加密身份包

**日期**: 2026-03-14

**決策**: 註冊時將 AES-256-GCM 加密的 privateKey 上傳 VPS。

**背景**: LINE WebView 的 localStorage 不可靠（LINE 關閉可能清除）。App 跟 WebView 的 storage 天生隔離。需要一個跨平台的身份存取方式。

**隱私分析**:
- VPS 存的是 AES-256-GCM 密文，沒有 PIN 無法解密
- Argon2id 抗 GPU/ASIC 暴力破解
- VPS 已經存了比這更敏感的資料（設備指紋、FCM token、風險分數）
- 加密身份包的隱私影響 = 零增量

**替代方案**: Google Drive（LINE WebView 裡 OAuth 體驗差）、QR 掃碼（需兩台裝置同時在手上）。VPS 是最通用的方案。

### ADR-009: 唯一臉部 ID — 確定性 hash 而非模糊匹配

**日期**: 2026-03-20

**決策**: 在現有 CNN 模糊匹配之外，新增確定性唯一 ID 路線。同一個人不管什麼條件掃臉，永遠產出相同的 hash。

**原因**:
- 防批量建號需要 O(1) 查重，模糊匹配需要 O(N) 遍歷
- 跨裝置恢復需要確定性身份，而非依賴保存的 embedding
- 去中心化驗證需要 hash 比對，不能依賴 embedding 比較
- AegisPay 等未來產品需要確定性的身份基礎設施

**技術路線**:
- 影像正規化: ArcFace align → gray → histEq → 橢圓遮罩 (SSIM 0.993 已驗證)
- 骨骼比率: 67 個 MediaPipe landmark 比率，篩選穩定子集 → 量化 → SHA-256
- pHash 4×4: 快速預篩 (100% exact match)
- 深度路線: 已放棄（穩定性 26%）

**與 CNN 的關係**: CNN 512 維 embedding 保留作為輔助驗證，不用於唯一 ID。

### ADR-010: 放棄深度路線 (Depth Anything V2)

**日期**: 2026-03-20

**決策**: 放棄使用單目深度估測作為唯一 ID 的特徵來源。

**原因**:
- 穩定性測試結果 12.3/48 (26%)，等於隨機水平
- Laplacian 曲率場是二階微分，推論誤差被放大而非消除
- 手機端 WASM 推論 ~12,600ms + OOM 風險
- bin width 0.002 太細，但根本問題是模型推論不確定性

### ADR-011: 骨骼比率大量擴充後篩選策略

**日期**: 2026-03-20

**決策**: 將骨骼比率從 12 個系統性擴充到 67 個，覆蓋所有面部區域，通過多條件穩定性測試篩選出 20-30 個穩定子集。

**原因**:
- 12 比率中 10 個穩定 (83%)，說明骨骼比率路線可行
- MediaPipe 有 468 個 landmarks，組合空間巨大，只用 12 個太浪費
- 不穩定的比率可以通過測試識別並淘汰
- 20 個穩定比率 × ~10 bins = 10^20 組合，遠超唯一 ID 需求

**測試計劃**: 頭不動 → 微轉頭 → 不同距離 → 不同光線 → 不同裝置，逐層淘汰。

### ADR-012: 聯合信心分數而非二元判斷

**日期**: 2026-03-14

**決策**: face 和 behavior 各自算相似度，加權為聯合信心分數，而非合成一個 hash 後做二元匹配。

**原因**: behavior 跨裝置有微小變異。如果合成一個 hash，behavior 的波動會把 face 的穩定性拖下水。分開評分可以容忍單一維度的變異。

### ADR-013: 統一骨骼比率系統取代雙軌 CNN+骨骼

**日期**: 2026-03-20

**決策**: 用骨骼比率一套系統同時服務註冊（3D 唯一 ID）和登入（平面 bin 匹配），不再使用 MobileFaceNet CNN 512 維 embedding。

**原因**:
- 兩套系統（CNN + 骨骼）增加複雜度和模型大小
- MobileFaceNet 13MB 模型變得不必要，MediaPipe FaceLandmarker ~4MB 即可
- 骨骼比率 24/24 穩定 (BIN_WIDTH=0.25, round())，3 輪測試完全一致
- 一套系統：註冊用 hash（確定性唯一 ID），登入用 bin similarity（≥80% match）
- 精簡客戶端體積，減少維護成本

**取捨**: 放棄 CNN 512 維的高維度區分力。但 24 骨骼比率 × ~5 bins = 5^24 ≈ 6×10^16 組合，遠超需求。

### ADR-014: 註冊時同時產出正面基準存本機

**日期**: 2026-03-20

**決策**: 3D 掃描時自動取最佳正面幀，算 2D 骨骼 bins，存本機作為日常登入比對基準。

**原因**:
- 日常登入只有平面刷臉，不可能每次轉頭
- 平面 vs 平面比對（正面基準 vs 登入幀）比 3D vs 平面更穩定
- 正面基準只存本機，不上傳 VPS（隱私保護）
- 登入閾值 ≥80% bin match（20/24 bins 一致）

### ADR-015: PIN 行為指紋用於註冊防雙胞胎

**日期**: 2026-03-20

**決策**: PIN 行為指紋與 3D 骨骼 hash 在註冊時聯合使用，主要目的是防雙胞胎。

**原因**:
- 同卵雙胞胎骨骼結構幾乎相同，骨骼 hash 可能一致
- 但 PIN 輸入的肌肉記憶（節奏、偏移、習慣）因人而異
- 聯合判定：骨骼 hash 相同 + PIN 行為不同 → 可能是雙胞胎 → 允許建立不同帳號
- 日常登入時 PIN 行為指紋作輔助驗證，不是主要判定

### ADR-016: 每個唯一 ID 綁定詐騙風險分數

**日期**: 2026-03-20

**決策**: 詐騙風險分數跟著唯一 Face ID 走，不跟裝置或帳號。

**原因**:
- 用唯一 ID 作為 key，同一張臉在任何裝置、任何 App 都是同一個風險分數
- 防止詐騙者換裝置「洗掉」風險記錄
- AegisPay 收款人信任判定：掃臉即可查風險分數
- 與 VPS `credit_scores` 表整合，key 從 LSH hash 改為 face_structure_hash

## 4. 待驗證假設

### 4.1 CNN 跨鏡頭穩定性

**假設**: MobileFaceNet 在不同前鏡頭下對同一張臉產出相似的 embedding。

**依據**: MobileFaceNet 的訓練數據集 (WebFace600K) 包含各種光線/角度/鏡頭的照片。

**風險**: 低端手機的前鏡頭品質可能太差，導致 embedding 偏移超過 LSH 容忍範圍。

**驗證方法**: Phase 2 實測。

### 4.2 PIN 18 維跨裝置穩定性

**假設**: 移除裝置相關特徵後，剩餘 18 維比率特徵在不同裝置上對同一人穩定。

**風險**: 不同螢幕大小可能影響手指移動速度，即使鍵盤固定尺寸。

**驗證方法**: Phase 3 實測。

### 4.3 LSH 閾值

**假設**: face LSH 128-bit、behavior LSH 64-bit、聯合信心分數 0.6/0.4 權重是最優配置。

**風險**: 可能需要根據實測調整。

**驗證方法**: Phase 15 校準。

## 5. 安全威脅模型

### 5.1 攻擊面

| 攻擊 | 防禦 |
|------|------|
| VPS 資料庫洩漏 | 三表分離 + HMAC → 無法關聯。blob 加密 → 無法解密 |
| 深偽影片 | anti-spoof 偵測 + bone bins 無法模擬 + PIN 行為 = 0 → 拒絕 |
| 雙胞胎 | 骨骼 hash 可能相同，但 PIN 行為不同 → 允許建不同帳號 |
| 批量建號機器人 | behavior CV 極低 + 無真實觸控面積 → 偵測 |
| 偷學 PIN 打法 | 學到 behavior 但 face 不同 → 拒絕 |
| 網路中間人 | HTTPS + blob 加密 → 即使攔截也無法解密 |
| 暴力破解 PIN | Argon2id (64MB memory, 3 iterations) → ~1 秒/次 → 100 萬組合需 11.5 天 |

### 5.2 我們有意不防禦的

| 場景 | 原因 |
|------|------|
| 同一人建 2 個帳號（隔 3 個月） | rate_limits 已過期。但 face_structure_hash 相同 → identity_anchors 會發現 |
| 用戶主動分享 PIN | 這是用戶自己的選擇，不是系統漏洞 |
| 超高品質 3D 面具 | 需要 MiniFASNet 防偽模型持續更新 |
