## 1. 專案定位

### 1.1 核心理念

**知道是同一人，但不知道是誰。**

AegisID 從 AegisTalk 的 Phase 10（PIN 行為指紋）、Phase 11（設備指紋）、Phase 12（信任分數）、Phase 26（CNN 臉部辨識）中抽離獨立，成為專責身份認證的獨立專案。

### 1.2 為什麼獨立

AegisTalk 是通訊 + AI 防詐。身份認證是跨產品的基礎設施，未來 AegisPay、AegisTrack 都需要身份層。獨立專案可以：

- 獨立版本管理和發布
- 作為 SDK 供多個產品使用
- 隱私設計有明確邊界

## 2. 架構決策記錄

### ADR-001: 身份由 FaceID + PIN 行為聯合構成

**日期**: 2026-03-14

**決策**: 使用 CNN FaceID (512維) 和 PIN 行為指紋 (18維) 共同代表一個人的身份，取聯合信心分數。

**原因**:
- 單獨 FaceID 無法防禦雙胞胎和深偽
- 單獨 PIN 行為無法跨裝置穩定辨識
- 兩者聯合 → 深偽通過 face 但沒有真人 PIN 行為 → 抓到
- 兩者聯合 → 雙胞胎通過 face 但 PIN 行為不同 → 抓到

**取捨**: 需要用戶同時刷臉和輸 PIN，增加一步操作。但這是安全和便利的合理取捨。

### ADR-002: 三表分離 + HMAC 隔離

**日期**: 2026-03-14

**決策**: VPS 使用三張獨立表，不同表使用不同 HMAC secret。

**考慮過的方案**:

| 方案 | 優點 | 缺點 |
|------|------|------|
| 單表全欄位 | 查詢簡單 | 交叉關聯 → 去匿名化風險 |
| 三表分離 + HMAC | 不可跨表關聯 | 查詢需多步 |
| 不存 VPS | 零風險 | 失去防濫用和跨裝置能力 |

**選擇三表分離的原因**:
- face 和 behavior hash 可關聯（都是抽象生物特徵），但不可關聯到 IP/裝置/pubkey
- rate_limits 48hr 自動清除，最壞情況只洩漏 2 天資料
- credit_scores 使用不同 HMAC secret，無法跟 identity_anchors JOIN

### ADR-003: PIN 行為指紋 18 維全比率設計

**日期**: 2026-03-14

**決策**: 移除所有裝置相關絕對值特徵，改為 18 維全比率/模式特徵。

**原因**: 分析現有 20 維特徵後發現，約 10+ 維是裝置指紋（觸控面積、力度、加速度計等），跨裝置完全不同。同一人換手機後 LSH hash 會變 → 無法跨裝置辨識。

**修正**: 只保留人的肌肉記憶特徵（節奏比率、空間偏移模式、習慣），配合固定尺寸鍵盤讓空間特徵也跨裝置穩定。

### ADR-004: 固定 PIN 鍵盤尺寸 280x360px

**日期**: 2026-03-14

**決策**: PIN 鍵盤固定為 280x360px，不隨螢幕縮放。

**原因**: 讓觸控位置的「鍵內偏移」成為人的特徵而非裝置的特徵。同一人在不同手機上按同一個鍵的相對偏移模式相同（肌肉記憶）。

### ADR-005: face_lsh_hash 和 behavior_lsh_hash 可關聯

**日期**: 2026-03-14

**決策**: identity_anchors 表同時存 face 和 behavior hash。

**隱私分析**:
- 可以關聯：face ↔ behavior（都是抽象生物特徵 hash）
- 不可關聯：face/behavior ↔ 真實身份（無 IP、無裝置、無 pubkey）
- 即使攻擊者知道某 face hash 和 behavior hash 屬於同一人，也無法知道這個人是誰

### ADR-006: IP 用完即丟（48hr TTL）

**日期**: 2026-03-14

**決策**: IP hash 只存在 rate_limits 表，48 小時自動清除。

**原因**: IP 對防批量建號有價值（同 IP 24hr 建 20 帳號 → 可疑）。但永久存儲 IP hash 有去匿名化風險（結合 ISP 記錄）。改為短期存儲，用完即丟。

**取捨**: 無法做 3 個月後的歷史追蹤。但詐騙集團批量建號是即時行為，48hr 窗口足夠。

### ADR-007: 只保留 CNN FaceID，廢棄 Landmark 版

**日期**: 2026-03-14

**決策**: 只使用 MobileFaceNet 512 維 CNN embedding，Landmark 128 維降級為 fallback。

**原因**:

| | Landmark 128維 | CNN 512維 |
|---|---|---|
| 跨鏡頭穩定性 | 中 | 高 |
| 抗光線/角度變化 | 低 | 高 |
| 額外模型 | 不需要 | 13MB |
| 計算速度 | <5ms | <50ms |

CNN 版本在所有維度都更強。Landmark 版本保留為 CNN 模型載入失敗時的 fallback。

### ADR-008: VPS 存加密身份包

**日期**: 2026-03-14

**決策**: 註冊時將 AES-256-GCM 加密的 privateKey 上傳 VPS。

**背景**: LINE WebView 的 localStorage 不可靠（LINE 關閉可能清除）。App 跟 WebView 的 storage 天生隔離。需要一個跨平台的身份存取方式。

**隱私分析**:
- VPS 存的是 AES-256-GCM 密文，沒有 PIN 無法解密
- Argon2id 抗 GPU/ASIC 暴力破解
- VPS 已經存了比這更敏感的資料（設備指紋、FCM token、風險分數）
- 加密身份包的隱私影響 = 零增量

**替代方案**: Google Drive（LINE WebView 裡 OAuth 體驗差）、QR 掃碼（需兩台裝置同時在手上）。VPS 是最通用的方案。

### ADR-009: 聯合信心分數而非二元判斷

**日期**: 2026-03-14

**決策**: face 和 behavior 各自算相似度，加權為聯合信心分數，而非合成一個 hash 後做二元匹配。

**原因**: behavior 跨裝置有微小變異。如果合成一個 hash，behavior 的波動會把 face 的穩定性拖下水。分開評分可以容忍單一維度的變異。

### AegisTalk 保留的部分

AegisTalk 仍保留：
- `webauthn.ts` — 系統生物辨識快捷解鎖（本地功能）
- `auth.ts` — PIN 驗證（本地功能）
- `keyEncryption.ts` — PIN 衍生加密金鑰（本地功能）

這些是 AegisTalk 的本地認證功能，不屬於 AegisID 的範疇。AegisID 負責的是跨裝置身份和 VPS 端的匿名驗證。

## 4. 待驗證假設

### 4.1 CNN 跨鏡頭穩定性

**假設**: MobileFaceNet 在不同前鏡頭下對同一張臉產出相似的 embedding。

**依據**: MobileFaceNet 的訓練數據集 (WebFace600K) 包含各種光線/角度/鏡頭的照片。

**風險**: 低端手機的前鏡頭品質可能太差，導致 embedding 偏移超過 LSH 容忍範圍。

**驗證方法**: Phase 2 實測。

### 4.2 PIN 18 維跨裝置穩定性

**假設**: 移除裝置相關特徵後，剩餘 18 維比率特徵在不同裝置上對同一人穩定。

**風險**: 不同螢幕大小可能影響手指移動速度，即使鍵盤固定尺寸。

**驗證方法**: Phase 3 實測。

### 4.3 LSH 閾值

**假設**: face LSH 128-bit、behavior LSH 64-bit、聯合信心分數 0.6/0.4 權重是最優配置。

**風險**: 可能需要根據實測調整。

**驗證方法**: Phase 15 校準。

## 5. 安全威脅模型

### 5.1 攻擊面

| 攻擊 | 防禦 |
|------|------|
| VPS 資料庫洩漏 | 三表分離 + HMAC → 無法關聯。blob 加密 → 無法解密 |
| 深偽影片 | face 通過但 behavior = 0 → 聯合分數 < 0.60 → 拒絕 |
| 雙胞胎 | face 通過但 behavior 不同 → 聯合分數 0.60~0.79 → 額外驗證 |
| 批量建號機器人 | behavior CV 極低 + 無真實觸控面積 → 偵測 |
| 偷學 PIN 打法 | 學到 behavior 但 face 不同 → 拒絕 |
| 網路中間人 | HTTPS + blob 加密 → 即使攔截也無法解密 |
| 暴力破解 PIN | Argon2id (64MB memory, 3 iterations) → ~1 秒/次 → 100 萬組合需 11.5 天 |

### 5.2 我們有意不防禦的

| 場景 | 原因 |
|------|------|
| 同一人建 2 個帳號（隔 3 個月） | rate_limits 已過期。但 face_lsh_hash 相同 → identity_anchors 會發現 |
| 用戶主動分享 PIN | 這是用戶自己的選擇，不是系統漏洞 |
| 超高品質 3D 面具 | 需要 MiniFASNet 防偽模型持續更新 |
